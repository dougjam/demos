# SPEC: Fire Sound Texture Synthesis Demo (Chadwick and James, SIGGRAPH 2011 &sect;5)

**Audience.** Claude Code, building a Python port plus a self-contained
interactive browser demo of the data-driven sound texture synthesis
algorithm.

**Goal.** Produce (a) a faithful Python port of the released C++ sound
texture synthesis code, (b) a deterministic JavaScript port that
reproduces the Python output to within FFT/floating-point round-off,
and (c) a single-page interactive browser demo bundling the five
released (base-signal, training-signal) examples, an "About this sound
model" disclosure, and the standard CS 448Z demo footer.

**Scope.** Sound texture synthesis only (Algorithm 2, &sect;5 of the
paper). Spectral bandwidth extension (&sect;4) is the sibling demo at
`../fire-bandwidth-extension/` (already shipped).

**Provided to you.**

- The paper PDF at `docs/2011 Chadwick - Fire Sound_rekt.pdf`.
- The original C++ source tree under `srcOrig/` (extracted from
  `sound_texture_synthesis.zip` at the canonical reference URL). This
  is the canonical implementation. Do not paraphrase it; port it.

**Canonical reference URL** (cite throughout):
<https://www.cs.cornell.edu/projects/Sound/fire/>

## 2. Python port

The Python port is the **ground truth** for the JS port. It is itself
verified against the C++ reference structurally (function-for-function,
data-flow-for-data-flow); bit-equivalence with the C++ is **not**
required, because the C++ uses ANN approximate NN search and has a
documented off-by-one in `buildGaussianLevel` (see `docs/ALGORITHM.md`).

### 2.1 Implementation rules

1. Read every file under `srcOrig/texture/` and `srcOrig/datastructure/`
   before writing any Python.
2. Translate C++ 0-based indexing to Python 0-based directly.
3. Use NumPy throughout. Use `scipy.spatial.cKDTree` for exact
   nearest-neighbour search. Do not vendor an alternate KD-tree.
4. **Pyramid construction.** 5-tap Gaussian filter
   `[0.05, 0.25, 0.40, 0.25, 0.05]`, 2:1 decimation. Padding to
   `(2^k + 1)` length, centred. Training pyramid: reflect the
   pre-pad-flank samples. Base pyramid: zero-pad the flanks.
5. **Off-by-one fix.** The C++
   `buildGaussianLevel` filter loop uses `j <= num_components` instead
   of `j < num_components`. This port uses `<`. Documented in
   `docs/ALGORITHM.md`.
6. **CDF init and sampling.** At top pyramid level, sort `|signal|`
   ascending into `_CDF`. `sampleCDF(fraction)` linearly interpolates
   indexed by `fraction * (N-1)`. `sampleInverseCDF(amplitude)` binary-
   searches the sorted array, returns the percentile.
7. **Feature vector layout.** For each output window at level `&ell;`,
   build a feature vector with two regions: causal (level &ell; samples
   to the **left** of the window center) and coarser-level (level
   &ell;+1 symmetric samples around the window center, linearly
   interpolated). Apply optional exponential `falloff` weight.
8. **Dynamic-range mapping.** When `scaleCDF=True` and the coarser
   level is the top level, rescale the coarser-level entries of the
   feature vector by `scaling = inputMagnitude / averageMagnitude`,
   blended with `(1-alpha) + alpha * scaling`. The matched window is
   then blended into the output with the inverse factor `1/scaling`.
9. **Hat-window blend.** Triangular window of half-width `windowHW`,
   summed at stride `windowHW` (so the windows COLA properly).
10. **Coarse-to-fine.** Synthesis loop runs from level
    `numLevels - 2` down to level `0`. The top level (`numLevels - 1`)
    is preserved from the input pyramid (i.e. only one base level is
    retained, matching the C++ default `numBaseLevels=1`).
11. **Exact NN.** All dictionary lookups use exact NN. The `epsANN`
    parameter is accepted but ignored; the slider in the UI mentions
    this explicitly.
12. **Determinism.** Same inputs &rArr; byte-identical output. The
    PCG32 RNG (shared with the JS port via `deterministic_rng.js`) is
    used only for tie-breaking in the rare case of identical feature
    vectors.

### 2.2 Public API

```python
def synthesize(
    base: np.ndarray,
    training: np.ndarray,
    fs: float = 44100.0,
    num_levels: int = 6,
    window_hw: int = 4,
    feature_hw: int = 3,
    falloff: float = 0.0,
    scale_cdf: bool = True,
    scaling_alpha: float = 1.0,
    rng: DeterministicRng | None = None,
    return_diagnostics: bool = False,
) -> np.ndarray | dict:
    """Faithful port of WindowSynthesizer.synthesizeSignal(). Returns
    the synthesised time-domain output (length = base.size). With
    return_diagnostics=True returns a dict with keys 'output', 'base_pyramid',
    'training_pyramid', 'output_pyramid', 'cdf_input', 'cdf_output',
    'scaling_per_window' (per spec §5.3 diagnostics)."""
```

## 3. Verification protocol (three tiers)

### 3.1 Tier 1: deterministic kernels

Each kernel has a fixed input + golden output. `atol=rtol=1e-13`.

| Test | Inputs | Golden |
|---|---|---|
| Pyramid level reduction | Synthetic input, 5-tap Gaussian | `gaussian_level_{name}.f64` |
| Pyramid level shapes | `padInputSignal` over a few sizes | scalar list in JSON |
| CDF sample / inverse | Hand-built sorted array | scalar in JSON |
| KD-tree determinism | Fixed-seed PCG32 input set, query a fixed point | scalar (matched index) in JSON |
| PCG32 first-N outputs | seed=42 | first 8 values inline |

### 3.2 Tier 2: phase-controlled full pipeline

For each of the 5 example pairs (`burning_brick`, `candle`, `dragon`,
`flame_jet`, `torch`), run the Python port deterministically and store
the synthesised waveform plus per-window scaling factors as compressed
`.npz` goldens.

Tolerance:
- `output`: byte-identical (`np.array_equal`) on Python re-runs.
- JS-vs-Python parity (run from `tests/validate.html`): peak abs error
  `< 1e-6`, peak relative error `< 1e-7`, RMS relative error `< 1e-9`.

The same goldens are the parity target for the JS port.

### 3.3 Tier 3: statistical smoke tests

Standalone Python checks (no goldens):

1. **Coarse-level preservation.** The bottom level of the output
   pyramid, when downsampled to the top level, equals the top level of
   the input pyramid bit-for-bit (since that level is preserved).
2. **Training PSD shape preservation.** Welch PSD of the synthesised
   output between 1 kHz and 10 kHz fits the training PSD slope within
   0.5 (in the absence of a CDF rescale that would change overall
   amplitude).
3. **CDF mapping helps amplitude match.** With `scale_cdf=True`, the
   output's amplitude CDF is closer to the input's than with
   `scale_cdf=False` (KS-distance check).
4. **Determinism.** Same inputs + same seed &rArr; byte-identical output.
5. **Pure-zero training.** Synthesizing with zero-amplitude training
   should produce zeros (defensive check).

## 4. JavaScript demo

### 4.1 Architecture

- Vanilla JS, no build step. Single-page app served by opening
  `index.html` over the local server (or via GitHub Pages).
- KD-tree: `kdtree.js` is a hand-port of
  `srcOrig/datastructure/KDTreeMulti.cpp`. Self-contained,
  deterministic, ~150 lines.
- DSP runs offline in a Web Worker. The worker is **bundled via Blob URL**
  in `main.js` (fetching `kdtree.js`, `deterministic_rng.js`,
  `texture_synthesis.js` with `cache: 'no-store'` so prior browser
  caching can't break us; the same pattern used by the sibling demo).
  `worker.js` exists as a standalone reference.
- 5 bundled (base, training) example pairs load on startup via
  `fetch()` from `assets/`.

### 4.2 Numerical agreement with Python

The JS port must reproduce the Python output within FFT-library round-
off on the validation case. The validation case (`tests/validate.html`)
uses a small synthetic (base = 4096-sample low-frequency burst,
training = 16384-sample colored noise) processed with default
parameters. Tolerance: max abs error `< 1e-6`.

Achieving this requires:

1. **Shared deterministic RNG** (PCG32, same seed &rArr; same stream).
2. **Identical feature-vector construction** (same formula, same loop
   bounds, same exponential falloff).
3. **Exact KD-tree search with identical tie-breaking**. The hand-
   ported KDTreeMulti uses `std::sort` semantics (Quicksort with
   median-of-3 in libstdc++); JS's `Array.prototype.sort` is required
   by spec to be stable but uses a different algorithm. Tie-breaking
   may therefore differ in the rare case of identical feature vectors.
   Document the achieved tolerance.

### 4.3 Asset format and pipeline

**headerless little-endian Float32** binaries (~12 MB total) under
`assets/{name}/`. The 6 raw Recordist WAVs at `srcOrig/work/*.wav`
(~5&ndash;15 MB total) copy unchanged into `assets/training_audio/`.

   | Slider | Range | Default | Notes |

### 4.5 Performance

- Largest example (`flame_jet`): base ~10 s &times; 44.1 kHz &asymp;
  441&times;10^3 samples; training similarly long. KD-tree with ~180&times;10^3
  features per top level.
- `scipy.spatial.cKDTree` builds in &lt; 200 ms per level.
- A hand-rolled JS KD-tree has higher per-step overhead but is still
  expected to build in 1&ndash;5 s per level.
- Per-window NN query: ~5&micro;s in cKDTree; ~50&micro;s in JS.
- Total wall time per processing run: 1&ndash;5 s in Python; 5&ndash;30 s
  in JS. If JS is too slow, **fall back** to precomputing per-level
  feature dictionaries server-side and shipping those under
  `assets/{name}/features_{level}.bin` plus a serialized KD-tree
  layout, which the JS only has to load and query.

## 5. Deterministic RNG (shared between Python and JS)

The same PCG32 used by the sibling demo. `python/deterministic_rng.py`
and `deterministic_rng.js` are re-used byte-for-byte (already
verified to produce identical first-1000 outputs at any seed). In this
demo the RNG is used only for tie-breaking in the rare case of two
identical feature vectors; same seed &rArr; same tie-breaks &rArr; same
output.
