# Test suite

Three tiers, per `SPEC_SpectralBandwidthExtension.md` section 3.

## Running

```bash
pip install numpy scipy pytest
cd <project root>
pytest python/tests/                       # all tests
pytest python/tests/test_unit.py           # tier 1 only
pytest python/tests/test_phase_controlled.py  # tier 2 only
pytest python/tests/test_statistical.py    # tier 3 only
```

## Tiers

### Tier 1 &mdash; deterministic kernels

`test_unit.py`, 9 cases. Six binary kernel goldens (`.f64` little-endian
Float64) plus a beta-quadratic scalar, plus three PCG32 PRNG checks
(first-8 outputs at seed 42, same-seed determinism, and uniform [0, 1)
range). Tolerance `atol=rtol=1e-13`. Runs in ~1.3 s.

### Tier 2 &mdash; full pipeline, phase-controlled

`test_phase_controlled.py`, 15 cases. For each of the 5 released test
signals (`burning_brick`, `candle`, `dragon`, `flame_jet`, `torch`)
crossed with `alpha &isin; {2.5, 3.0, 3.5}`, the test rebuilds the
seed-42 PCG32 phase array, runs `extend_signal` with `phases_override`,
and compares the extended waveform and per-window beta values against
compressed `.npz` goldens. Tolerance per spec section 3.2:

- `betas`: `atol=1e-12, rtol=1e-10`
- `extended`: peak relative error `< 1e-10` and L2 relative error `< 1e-12`

This tier is slow because every per-window FFT runs at the full
Matlab-faithful NFFT (262144 or 524288 depending on signal length). On
this Python implementation a full pass is ~21 min: ~30&ndash;40 s per
burning_brick / torch case (NFFT = 262144) and ~85&ndash;145 s per
candle / dragon / flame_jet case (NFFT = 524288). Goldens are committed;
the test just re-runs and compares.

### Tier 3 &mdash; statistical smoke tests

`test_statistical.py`. No goldens; check expected output properties:

- COLA identity at Nyquist cutoff
- silent input
- sub-cutoff sine preservation
- log-log slope above cutoff for `alpha &isin; {2.0, 2.5, 3.0, 3.5}`
- envelope sync via Hilbert correlation > 0.7
- byte-identical determinism for fixed seed

## Goldens

`golden/` contains binary fixtures plus `tier1_metadata.json` /
`tier2_metadata.json` describing the inputs that produced each. They are
**generated by the Python port itself** (the canonical implementation in
this repo) via `python/tools/generate_goldens.py`. Reasons:

1. The released Matlab is the reference for the line-by-line port (see
   `srcOrig/matlab/` and `docs/ALGORITHM.md`), but the goldens here pin
   the **Python port** against accidental regression. Any change to the
   Python kernels that perturbs output by more than the documented
   tolerance will fail Tier 1 immediately.
2. The Tier 2 goldens are the **parity target** that the JavaScript
   port must reproduce. The JS port consumes the same shared PCG32
   stream and must produce numerically identical extended waveforms
   (subject to FFT-library round-off, see `tests/validate.html`).

If the Python port&apos;s output ever needs to be re-pinned (e.g., a
deliberate algorithmic change), regenerate with:

```bash
python python/tools/generate_goldens.py            # both tiers
python python/tools/generate_goldens.py --tier 1   # just kernels
```

The bytes-vs-Matlab tolerance is **not measured** in this configuration
(no Octave/Matlab in the toolchain). The Python port follows the Matlab
source line-by-line and was sanity-checked by reading every `.m` file
before porting; the per-rule fidelity decisions are documented in
`docs/ALGORITHM.md`.
