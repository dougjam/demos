# SPEC: Spectral Bandwidth Extension Demo (Chadwick and James, SIGGRAPH 2011)

**Audience.** Claude Code, building a Python port plus a self-contained interactive browser demo.

**Goal.** Produce (a) a faithful Python port of the released Matlab spectral bandwidth extension code, verified against the Matlab reference to within numerical precision on the provided test cases, and (b) a single-page JavaScript demo that ships with the five released test signals embedded as bundled assets, lets the user play the original and the extended sound, and adjusts the algorithm parameters interactively via sliders.

**Scope.** Spectral bandwidth extension only (Algorithm 1, §4 of the paper). Not the data-driven sound texture synthesis from §5.

**Provided to you.**
- The paper PDF at `docs/2011 Chadwick - Fire Sound_rekt.pdf`.
- The original Matlab source tree in `srcOrig/`. This contains the canonical implementation. Do not paraphrase it; port it.

https://www.cs.cornell.edu/projects/Sound/fire/

## 2. Python port

The Python port is the **ground truth** for the JS port. It is itself verified against the Matlab reference. Do not attempt to "improve" the algorithm; port it exactly.

### 2.1 Implementation rules

1. Read every file in `srcOrig/matlab/` before writing any Python.
2. Translate Matlab 1-based indexing to Python 0-based carefully and only at the boundary; internally, mirror the Matlab indexing scheme to keep the translation auditable.
3. Use `numpy.fft.fft` / `numpy.fft.ifft` (two-sided), not `rfft`/`irfft`. The Matlab uses two-sided and the integration in `fit_dual_power_spectra.m` runs over the full two-sided axis. Switching to one-sided would give numerically different `beta` values.
4. `NFFT = 2 ** ceil(log2(L))` where `L` is the full signal length (Matlab `nextpow2`). Per-window FFTs zero-pad the windowed slice into a length-`NFFT` buffer.
5. The lowpass for the envelope is `scipy.signal.butter(2, fc/(fs/2), 'low')` plus `scipy.signal.filtfilt`. This matches `lowpass_filter.m` exactly because `filtfilt` is the standard zero-phase implementation.
6. The triangular window has peak value 1 (not `halfWidth`); adjacent windows COLA to 1, so no final divide. This is consistent with the Matlab `build_window_function_linear.m`.
7. The power-law noise is generated **once** before the window loop, with the high-pass blend filter `blend2` applied in the frequency domain before the IFFT. The same time-domain noise array is reused across all windows. This is what makes adjacent windows produce coherent (overlapping-add-friendly) noise rather than independent draws.
8. The envelope multiplier is `|p_lowpass| * w_i`, not `|p * w_i|` and not `|p|`. The lowpass-filtered version of the input is what shapes the noise amplitude.
9. The `beta` quadratic in `fit_dual_power_spectra.m` uses 6 trapezoidal integrals (real and imaginary parts of `Y_L`, `Y_N` separately) plus one for `|Y_S|^2`, then solves `a beta^2 + b beta + c = 0` with the textbook stable form (Vieta's identity, branched on the sign of `b`). The two roots have opposite sign when `c < 0` (the usual case); pick the positive one with `max(beta1, beta2)`.
10. `Y_S` (the target spectrum in the quadratic) is the **unfiltered** windowed signal `Y = FFT(psub)`, not `Y_L`. Match the Matlab.
11. There is a `% FIXME` comment in `build_powerlaw_spectrum.m` about whether the magnitude should be `x^exponent` or `(2*pi*x)^exponent`. The released code uses `x^exponent`. The constant cancels in `beta`, so the audible output is identical. Match the released form.

### 2.2 Public API

        'extended', 'lowpass', 'noise_full', 'filtered', 'betas'

## 3. Verification protocol (three tiers)

### 3.1 Tier 1: deterministic kernels (bit-exact)

These have no randomness and must agree with the Matlab to within IEEE-754 round-off (about `1e-14` relative).

For each, write a small Octave script in `python/tools/octave_run.m` that produces a binary `.f64` golden file in `python/tests/golden/`. The Python test loads the golden and compares.

| Test | Octave call | Golden file |
|---|---|---|
| Triangular window | `build_window_function_linear(500, 1024)` | `window_500_L1024.f64` |
| Blend filters | `build_blending_function_linear(x, NFFT, 165, 195)` for fixed `x`, `NFFT` | `blend1.f64`, `blend2.f64` |
| Gaussian weight | `build_blurring_function_gaussian(x, 180, 30, NFFT)` | `gaussian.f64` |
| Powerlaw spectrum (magnitudes only) | `abs(build_powerlaw_spectrum(-1.25, x, NFFT))` | `powerlaw_mag.f64` |
| Beta quadratic | `fit_dual_power_spectra(...)` on a hand-constructed input | scalar in JSON |
| Lowpass filter | `lowpass_filter(p, 44100, 180)` for a fixed synthetic `p` | `lowpass.f64` |

Tolerance: `np.allclose(py, golden, atol=1e-13, rtol=1e-13)`.

### 3.2 Tier 2: phase-controlled full pipeline (bit-exact)

The only nondeterminism in `extend_signal` is the random phases drawn inside `build_powerlaw_spectrum`. To get bit-exact agreement with Matlab, **save those phases from a single Matlab run and have Python load them**.

Modify a copy of `srcOrig/matlab/build_powerlaw_spectrum.m` (placed in `python/tools/octave_run.m` as a local override) to write the random phase array to `phases_seed{N}.f64` before using it. Equivalently, generate phases in Octave with `rand('state', N); phases = 2*pi*rand(NFFT, 1);` and pass them in via a small wrapper.

The Python `extend_signal` then accepts an optional `phases_override: np.ndarray` argument. When supplied, it skips the RNG call and uses the provided phases directly.

For each of the 5 test signals, with `alpha in {2.5, 3.0, 3.5}` and a fixed phase array (`phases_seed42.f64`), produce:
- `{name}_extended_alpha{a}_seed42.f64`: full output waveform
- `{name}_betas_alpha{a}_seed42.f64`: per-window beta values

Total: 15 cases × 2 files = 30 golden files. Storage is modest (each waveform is at most ~10 s × 44.1 kHz × 8 bytes = 3.5 MB; ~50 MB total; acceptable for a research repo, but optionally compress with `np.savez_compressed`).

Pytest parametrizes over cases. Tolerance:
- `betas`: `atol=1e-12, rtol=1e-10` (a scalar from a quadratic solve).
- `extended`: peak relative error `< 1e-10` and L2 relative error `< 1e-12`. (Some tolerance for FFT library roundoff differences.)

If exact equivalence cannot be reached due to FFT library differences (numpy uses pocketfft, Matlab/Octave use FFTW), document the actual tolerance achieved and require it stay there. In practice numpy and FFTW agree to ~`1e-13` on these sizes.

### 3.3 Tier 3: statistical smoke tests (no Matlab dependency)

Run the Python port standalone and check expected properties of the output. These don't require Matlab/Octave on the test machine.

1. **COLA identity.** With `f_cutoff` set to the Nyquist frequency (so `blend1 ≡ 1` everywhere on the positive axis), `blend2 ≡ 0`, and the noise contribution vanishes. Output should equal the input under COLA. Tolerance: `1e-10` relative RMS.
2. **Silent input.** `p = zeros(N)`. Output exactly zero.
3. **Sub-cutoff preservation.** Input is a 100 Hz sine. Magnitude at 100 Hz in the output matches the input within 0.5 dB.
4. **Power-law slope above cutoff.** Input is filtered white noise band-limited below 150 Hz. For `alpha in {2.0, 2.5, 3.0, 3.5}`: Welch PSD of output between 500 Hz and 10 kHz, fit log-log slope, assert within 0.3 of `-alpha`.
5. **Envelope synchronization.** 60 Hz carrier amplitude-modulated by a slow on/off envelope. Hilbert envelope of high-passed (`> 500 Hz`) output correlates with the modulating envelope of the input at Pearson `> 0.7`.
6. **Determinism.** Same input, same `rng` seed, same parameters: output is byte-identical across runs.

## 4. JavaScript demo

### 4.1 Architecture

- Vanilla JS, no build step. Single-page app served by opening `index.html` directly or via any static server (or via GitHub Pages from the demos repo).
- FFTs: vendor `fft.js` (https://github.com/indutny/fft.js, MIT) at the project root and keep its license header intact.
- DSP runs offline (in a Web Worker for responsiveness on the longer 10-second clips). Web Audio API is used only for playback. Output is rendered to an `AudioBuffer`.
- All 5 example signals load on startup via `fetch()` from `assets/` (relative to `index.html`).

### 4.2 Numerical agreement with Python

The JS port must reproduce the Python output to within FFT-library round-off on the validation case. Achieving this requires:

1. **A shared deterministic RNG.** Both `python/deterministic_rng.py` and `deterministic_rng.js` (at the project root) implement the same PRNG, e.g., a 32-bit xorshift or PCG-XSH-RR with a documented constant. Same seed yields the same uniform `[0, 1)` sequence in both languages, byte-for-byte (verify with a unit test that compares the first 1000 outputs).
2. **The same per-window indexing**, exactly mirroring the Matlab/Python loop.
3. **Real-FFT consistency.** `fft.js` uses two-sided complex FFTs and matches numpy's `fft.fft` to within ~`1e-12` on the relevant sizes. Verify with a small test using a known input.

Validation case in `tests/validate.html` (at the project root): a hard-coded short synthetic input (e.g., 4096 samples of a low-frequency burst), processed with a fixed seed and default parameters. The expected output (a Float32 array) is bundled as `tests/golden_synthetic.json`, generated by the Python port. The validation page loads this, runs the JS port, and reports max abs error. Threshold: `< 1e-6` for Float32 storage, with a console log of the actual achieved value.

### 4.3 Asset format and pipeline

The 5 `.vector` files total ~12 MB at float64. Convert to **headerless little-endian Float32** binaries (5.7 MB total) under `assets/` at the project root. Run from the project root:

```python
# python/tools/vectors_to_assets.py
# Run from the project root: python python/tools/vectors_to_assets.py
import numpy as np, json, sys
from pathlib import Path
sys.path.insert(0, str(Path(__file__).resolve().parents[1]))   # add python/ to path
from vector_io import read_vector

ROOT = Path(__file__).resolve().parents[2]   # project root
src = ROOT / 'srcOrig' / 'work'
dst = ROOT / 'assets'
dst.mkdir(parents=True, exist_ok=True)

manifest = {'sample_rate': 44100, 'format': 'float32_le', 'examples': []}
for vf in sorted(src.glob('*.vector')):
    name = vf.stem
    data = read_vector(vf).astype(np.float32)
    (dst / f'{name}.bin').write_bytes(data.tobytes())
    manifest['examples'].append({
        'name': name,
        'file': f'{name}.bin',
        'samples': int(data.size),
        'duration_s': float(data.size / 44100),
    })

(dst / 'manifest.json').write_text(json.dumps(manifest, indent=2))
```

Loaded in JS as:

```javascript
const buf = await (await fetch(`assets/${name}.bin`)).arrayBuffer();
const samples = new Float32Array(buf);
```

(Float32 is fine here; the original signal's dynamic range is well within Float32 precision and the FFT pipeline runs in Float32 anyway.)

### 4.5 Performance

- 10 seconds at 44.1 kHz is 441,000 samples. NFFT = 524,288. Per-window FFT work is dominated by NFFT-size FFTs, of which there are about `L / halfWidth` ≈ 880 with the default `halfWidth = 500`. Total: ~1,760 length-524,288 FFTs (one for the signal, one for the noise per window). At ~50 ms per FFT in pure JS, this is ~90 s, which is too slow for an interactive demo.
- **Optimization required:** zero-pad each per-window FFT to a smaller `NFFT_window`, e.g. `next_pow2(8 * halfWidth) = 4096` or `8192`. This reduces the FFT cost by ~64x at the cost of slightly different `beta` values (the trapezoidal integral is over a coarser frequency grid). Document the tradeoff in `docs/ALGORITHM.md` and add a slider toggle "Faithful FFT length (slow)" vs "Demo FFT length (fast)" defaulting to fast.
- The Python port should also expose this as a parameter (`nfft_per_window: int | None`, default `None` meaning the Matlab-faithful full-signal-length FFT). Tier 2 tests use `None` (full faithfulness); Tier 3 statistical tests can use the smaller value.
- If the smaller FFT length materially changes the audible output, fall back to the full length and use a Web Worker to keep the UI responsive.

## 5. Deterministic RNG (shared between Python and JS)

A single small PRNG implementation, ported byte-for-byte. Recommended: PCG32 (https://www.pcg-random.org/), which has a clean C reference and trivial implementations in any language. Alternatively a 64-bit xorshift. The exact choice doesn't matter; what matters is bit-equivalence between the two languages, verified by a unit test that compares the first 1000 outputs of a fixed-seeded sequence.

```python
# python/deterministic_rng.py
class DeterministicRng:
    def __init__(self, seed: int): ...
    def random(self) -> float: """Returns uniform [0, 1)."""
    def random_array(self, n: int) -> np.ndarray: ...
```

```javascript
// deterministic_rng.js   (project root)
class DeterministicRng {
    constructor(seed) { ... }
    random() { /* returns uniform [0, 1) */ }
    randomArray(n) { /* Float32Array */ }
}
```

For the Tier 2 phase-controlled tests, **don't use this PRNG**; instead load the Octave-generated phase array directly. The shared PRNG is only used for casual use of the Python port and for the JS demo.
